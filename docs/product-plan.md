# 足球盘口预测平台产品方案

> 版本：V2.0
> 状态：前瞻数据积累与标准化数据库建设
> 更新日期：2026-07-17

## 1. 产品定义

产品定位为：基于纯盘口数据的赛前概率预测、临场监控与赛后复盘平台。

采集范围采用“500 足球竞彩有什么比赛就采什么”的策略，不设置联赛白名单。五大联赛、欧战、世界杯及预选赛、韩国 K1 继续作为重点评估分组，但不限制其他赛事进入原始数据层。

建模采用“广泛采集、分层准入、选择性建模”：只有盘口完整、时间合格、身份无冲突且具有明确 90 分钟赛果的数据才能进入严格训练或回测。

所有预测针对常规时间 90 分钟及补时，不包含加时赛和点球大战。产品不输出投注金额、投注组合、成本、收益、ROI 或回报承诺。

## 2. 数据与模型边界

长期保存：

- 500 足球竞彩比赛清单、销售状态、官方让球及各玩法 SP。
- 多公司欧赔、亚洲让球、大小球和让球指数。
- 原始响应、来源 URL、HTTP 元数据、内容哈希和解析版本。
- 完场候选比分和经确认的 90 分钟赛果。
- 缺失、失败、迟到、身份冲突、结构变化和赛果歧义。

V1 模型只使用欧赔、亚洲让球、大小球、多公司盘口变化及相关时间信息。竞彩官方 SP 和让球指数先保存，未形成独立决策前不作为 V1 模型输入。

根据 D-023，盘口采用追加式标准化 V2。欧赔继续解析来源工作簿；亚盘、大小球和让球指数直接解析正确解码的 HTML。亚洲盘统一以主队视角表示，主让为负、受让为正；大小球分盘取两端均值。采集层成功不等于模型字段完整，只有三个核心市场各有至少 3 家不同且字段完整的 bookmaker，批次才可能进入模型严格资格。

历史回抓可用于字段和覆盖研究，但必须标记 `backfill=true`、`strict_backtest_eligible=false`。严格回测只使用本系统真实前瞻观察的数据。

根据 D-022，项目允许在完全隔离的研究层使用公开静态历史数据运行离线市场基线。该数据同时标记 `research_only=true` 和 `cutoff_eligible=false`，只回答“纯盘口方法是否值得继续研究”，不得生成当前比赛预测、进入正式模型或计入阶段 4 门禁。

## 3. 核心输出

每场比赛在 `T-24h`、`T-6h`、`T-60m`、`T-10m` 生成不可覆盖的预测版本。缺少合格快照时降级或停止发布。

| 输出 | 定义 |
| --- | --- |
| 胜平负 | 去除返还率并校准后的概率 |
| 预测结论 | 主胜、平局、客胜或双重机会倾向 |
| 市场隐含进球 | 由欧赔、让球和大小球联合反演，不称为真实 xG |
| 比分点 | 概率最高的三个常规时间比分，低权重展示 |
| 总进球 | `0-1`、`2-3`、`4+` 概率 |
| 双方进球 | Yes/No 概率 |
| 市场状态 | 共识、分歧、反转或异常 |
| 置信等级 | 基于数据质量、样本外程度和历史校准 |
| 追溯信息 | 截止时间、数据版本、模型版本和风险标签 |

## 4. 数据准入与时间规则

原始响应 append-only，同内容可按 SHA-256 去重，但每次观察保留独立 manifest。每条公司盘口行保存：

```text
source_event_time
observed_at
ingested_at
corrected_at
```

`observed_at` 是 HTTP 响应完整接收时间。预测和回测只能使用 `observed_at <= prediction_cutoff` 的记录。供应商事后修正不得改写已经发布的历史预测。

必须分开统计程序采集失败、解析失败、来源缺盘和严格切点不合格，不能把小赛事没有某类盘口误报为程序故障。

V1 原始标准化记录永久保留。由原始 blob 和 manifest 离线重放得到的 V2 记录标记 `event_origin=reprocess`，只修正派生解析结果，不改变来源观察时间，也不进入实时连续验收指标。

## 5. 数据处理与模型体系

1. 保存不可变原始层和可重放 manifest。
2. 标准化赛事、球队、公司、赔率格式、盘口线和时间。
3. 对欧赔、让球和大小球分别去除返还率。
4. 使用稳健中位数形成同切点市场共识。
5. 联合拟合三类市场，估算主客进球参数并生成比分矩阵。
6. 只有基线稳定后，才使用 LightGBM 等模型学习盘口变化、公司分歧和跨市场一致性。

增强模型必须同时具备至少 6 个月时间跨度和 3,000 场严格样本。单一赛事评估组少于 200 场时不得单独校准或输出高置信结论。

## 6. 赛果与回测

自动采集的比分先形成 `ResultCandidate`。只有能够证明为常规时间 90 分钟比分的数据才生成 `VerifiedResult`。可能加时或赛事格式未知的比赛默认不进入严格训练。

根据 D-019，赛果闭环不设置人工确认步骤。日期直播页的唯一完场 fixture 行可形成候选；直播页与分析页比分一致、身份无冲突且赛事登记为 `regular_time_only` 时自动确认。可能加时或未知赛事自动隔离，允许永久不进入训练，不能为了提高覆盖率无条件贴上 90 分钟标签。

回测统一使用时间滚动验证，禁止随机拆分。主要指标为 Log Loss、Brier Score、RPS、ECE、可靠性曲线和分布 Log Loss；比分命中只作辅助指标。每个赛事和预测切点必须分别报告样本量与结果。

## 7. 产品能力

- 比赛中心：按日期、赛事、状态、预测方向、置信度和风险筛选。
- 单场分析：盘口时间线、公司分歧、隐含进球和冻结预测版本。
- 临场监控：异动、冲突和方向变化。
- 预测档案：不可覆盖的历史预测。
- 检查复盘：90 分钟赛果、预测偏差和异常证据。
- 模型仪表盘：回测、校准、样本量和版本。
- 数据质量中心：发现失败、来源缺盘、解析失败、迟到、映射冲突和采集空窗。

## 8. 技术路线与阶段门禁

验证采集器采用 Python、不可变文件存储、标准化 JSONL 和可重建 SQLite 调度状态。分析层采用 PostgreSQL，但数据库只能从文件事实层重建，不得替代原始 blob、manifest 和 JSONL。后续产品使用 FastAPI、React 和 ECharts。

根据 D-015，项目负责人已授权在 24 小时、7 天和 30 天采集验收继续运行时提前建设阶段 3 数据库；未完成的采集验收仍然约束生产可用声明。数据库重放和 as-of 防泄漏验收完成前不开发模型，模型门禁完成前不开发 Web 产品。阶段顺序和完成条件以 `docs/execution-plan.md` 为准。

根据 D-016、D-017、D-018，项目已使用实际创建的阿里云杭州 Ubuntu 22.04 单台 ECS 完成隔离 smoke，但暂缓购买正式数据盘并暂停云端正式迁移。40 GB 系统盘不得承载长期事实层；没有至少 100 GB 数据盘和 OSS 恢复闭环前，不启用云端长期 timer，不停止 Windows 本地采集。迁移不改变数据和模型门禁，未来云端正式切换后 7 天技术验收和 30 天稳定性验收重新计时，本地与云端运行时间不得拼接。

根据 D-020 和 D-021，本地长期运行使用 G 盘双层异盘备份：每日增量镜像用于快速恢复，每周内容寻址批次用于 SHA-256 审计。备份必须从锁定时刻的一致性文件清单和 SQLite 快照生成，PostgreSQL 继续以事实文件空库重放作为恢复标准。采集和两个备份任务使用 S4U；数据库任务在当前独立 Windows 主机使用专用非管理员账户及 Task Scheduler 加密保存的随机凭据。四个任务最终均须通过注销及重启验证。

根据 D-023，盘口 V2 的 7 天技术子窗口和 30 天切点资格子窗口从 V2 正式部署时重新计时；已通过的 24 小时发现结论和正在累计的赛果子窗口不重置。整体 30 天验收必须等待所有子窗口分别通过。

阶段 4 的 500 场门禁按不同预测切点分别统计不同 fixture，不得把同场多个切点相加凑数。某个切点达到 500 场严格快照及唯一有效已验证赛果后，只允许先建设该切点的市场基线。

历史研究层使用 `data/research/` 和 PostgreSQL `research` schema，与正式 `football` schema、导入检查点和资格视图完全分离。首版只比较均匀概率、赛事历史先验和去水市场概率，不使用复杂增强模型，不把2026回放称为盲测或严格样本外。
