## D-001：V1 只使用盘口市场数据

- 日期：2026-07-15
- 状态：已接受
- 背景：需要验证仅依靠市场信息能否形成可校准、可复现的概率预测。
- 决定：V1 输入限定为欧赔、亚洲让球、大小球、多公司盘口变化和相关时间信息。
- 理由：控制变量和数据范围，优先解决数据质量、时间语义及防泄漏问题。
- 影响：球队新闻、伤停、阵容、天气、舆情和人工观点不进入 V1 模型。
- 替代方案：多源球队基本面模型；留待 V1 基线稳定后另行决策。

## D-002：预测口径为常规时间 90 分钟及补时

- 日期：2026-07-15
- 状态：已接受
- 背景：杯赛可能包含加时和点球，必须统一标签口径。
- 决定：所有预测和结算默认只计算常规时间 90 分钟及补时。
- 理由：与主流 1X2 常规时间市场保持一致，避免混合不同结算规则。
- 影响：加时和点球不得用于胜平负标签；赛果源必须能区分常规时间结果。
- 替代方案：预测晋级结果；不属于当前产品范围。

## D-003：使用四个固定预测切点

- 日期：2026-07-15
- 状态：已接受
- 背景：需要比较不同临场阶段并冻结可追溯版本。
- 决定：使用 `T-24h`、`T-6h`、`T-60m`、`T-10m` 四个切点。
- 理由：兼顾早期判断、赛前变化和临场状态。
- 影响：采集频率、新鲜度、快照和回测均围绕四个切点设计。
- 替代方案：连续实时预测；首版复杂度过高，暂不采用。

## D-004：原始数据 append-only 并保留四类时间

- 日期：2026-07-15
- 状态：已接受
- 背景：供应商可能更新或事后修正盘口记录，需要保留完整历史。
- 决定：原始响应只追加，并保存 `source_event_time`、`observed_at`、`ingested_at`、`corrected_at`。
- 理由：支持审计、重放和识别事后修正。
- 影响：供应商修正不能覆盖原始记录或已经发布的预测。
- 替代方案：只保存最新状态；无法可靠复现，拒绝采用。

## D-005：采用严格的 observed-at 防泄漏规则

- 日期：2026-07-15
- 状态：已接受
- 背景：数据源声明时间可能早于系统实际获得数据的时间。
- 决定：预测和回测只能使用 `observed_at <= prediction_cutoff` 的数据。
- 理由：确保离线回测不会使用当时系统尚未看到的信息。
- 影响：所有快照、特征、公司权重和训练切分都必须接受时间审计。
- 替代方案：只按供应商事件时间过滤；存在未来数据泄漏风险，拒绝采用。

## D-006：MVP 基础技术栈

- 日期：2026-07-15
- 状态：已接受
- 背景：项目需要可维护的采集、建模、API 和可视化能力。
- 决定：MVP 采用 Python、FastAPI、PostgreSQL、独立采集 Worker、React、ECharts 和 Docker Compose。
- 理由：技术成熟，适合单用户私有系统，并能覆盖数据和 Web 工作流。
- 影响：暂不默认引入 TimescaleDB、Redis 或 Celery，达到明确需求后再决策。
- 替代方案：单体脚本或更复杂的分布式架构；均不作为 MVP 默认方案。

## D-007：预算、部署地区和告警方式

- 日期：2026-07-15
- 状态：部分替代，部署地区被 D-016 替代
- 背景：执行计划需要一个可估算的运行环境，但尚未获得最终人工确认和供应商报价。
- 决定：暂按每月不超过 CNY 300、香港或新加坡 Ubuntu VPS、email 告警规划。
- 理由：为早期方案提供保守默认值。
- 影响：这些值不能作为采购授权，也不能用于降低数据验收标准。
- 替代方案：本地常驻、其他云地区或其他告警渠道；获得实际需求后再比较。

## D-008：当前采用单总控 Agent

- 日期：2026-07-15
- 状态：已接受
- 背景：项目仍在数据源调研期，立即维护多个角色 Agent 会增加交接和同步成本。
- 决定：使用根目录 `AGENTS.md` 作为单总控 Agent 契约，按项目阶段切换数据审计、数据工程、模型验证和产品运维职责。
- 理由：保持唯一状态入口和阶段门禁，同时为后续拆分保留清晰职责边界。
- 影响：当前不创建角色专属 Agent 文件或自定义 Skill。
- 替代方案：立即建立多角色 Agent；待并行开发需求实际出现时重新评估。

## D-009：个人研究数据获取范围与来源组合

- 日期：2026-07-15
- 状态：已接受，主数据源定位被 D-011 部分替代
- 背景：用户希望在不考虑商业授权和收费的个人学习研究前提下，先获取 2025 年至当前时间的数据进行探索。
- 决定：研究数据范围为 `2025-01-01T00:00:00Z` 至每次执行时的真实时间；免费来源组合暂定为 Football-Data、OddsHarvester 和 FlashscoreScraper，具体步骤见 `docs/research-data-acquisition-plan.md`。
- 理由：该组合能覆盖赛果、开盘/收盘基线、历史盘口研究和少量人工交叉验证，适合先验证字段结构和覆盖风险。
- 影响：可以执行范围受限的研究数据下载和小样本验证；该决定不代表正式数据源授权验收通过，也不允许跳过阶段 1。
- 替代方案：直接购买授权 API；在正式产品验收中仍需评估，但当前研究阶段暂不执行。

## D-010：历史回抓数据与严格回测资格分层

- 日期：2026-07-15
- 状态：已接受
- 背景：历史回抓数据的真实观察时间晚于比赛发生时间，若直接用于 observed-at 回测会产生未来数据泄漏。
- 决定：历史回抓数据必须标记 `backfill=true`，`observed_at` 使用真实下载时间，`strict_backtest_eligible=false`；只有从现在开始前瞻采集且满足防泄漏条件的数据才可标记为严格回测可用。
- 理由：保留历史研究价值，同时不破坏项目的 observed-at 防泄漏原则。
- 影响：历史回抓数据可用于覆盖研究、字段验证和非严格分析，但不能用于声称严格模拟当时可见信息的回测。
- 替代方案：按数据源声明时间直接参与严格回测；存在泄漏风险，拒绝采用。

## D-011：500 作为前瞻研究主数据源

- 日期：2026-07-15
- 状态：已接受
- 背景：500 竞彩页面能够发现带稳定 fixture ID 的比赛，500 分析页已验证可取得欧赔、亚盘、大小球和让球指数；用户愿意从当前时间连续积累 1 至 2 年。
- 决定：个人研究阶段使用 500 作为前瞻采集主数据源，付费或授权 API 调研降为未来可选项；本决定部分替代 D-009 的主来源定位，但不删除既有历史研究数据。
- 理由：前瞻真实观察能够建立严格 `observed_at` 时间线，并避免依赖不完整历史回填。
- 影响：当前阶段门禁改为 500 技术验收；7 天和 30 天验收通过前仍不开发数据库、模型或 Web 产品。
- 替代方案：继续等待 2 至 4 个授权 API；不再作为当前阻塞。

## D-012：采集全部竞彩比赛并分层准入

- 日期：2026-07-15
- 状态：已接受
- 背景：固定赛事白名单会漏掉竞彩页面上的其他比赛，也降低长期样本积累速度。
- 决定：采集六个足球竞彩玩法页面中所有 fixture ID 的并集，不设联赛白名单；训练、回测和发布资格按盘口、时间、身份和赛果质量单独判断。
- 理由：数据保存范围和模型适用范围是不同问题，广泛保存不会自动授权低质量比赛进入模型。
- 影响：所有质量指标按赛事分组；来源缺盘与程序失败必须分开统计。
- 替代方案：只采五大联赛、欧战、世界杯和 K1；拒绝作为采集边界，保留为重点评估分组。

## D-013：本地 Windows 完成首轮运行验证

- 日期：2026-07-15
- 状态：部分替代，云端准备见 D-016
- 背景：当前开发环境和既有数据均位于 Windows，本阶段优先验证数据闭环而非部署架构。
- 决定：首版使用本地 Windows 任务计划程序运行，完成 7 天技术验收和 30 天稳定性验收后再评估 VPS。
- 理由：复用现有环境，缩短开始前瞻积累的时间。
- 影响：人工必须保证联网、不休眠和时钟准确；30 天验收前必须配置异盘或网络备份。
- 替代方案：从第一天部署 Linux VPS；留待季度复盘。

## D-014：保存竞彩 SP 但不纳入 V1 模型

- 日期：2026-07-15
- 状态：已接受
- 背景：竞彩发现页面同时提供官方让球、销售状态和多种玩法 SP，长期保存成本较低，但会扩大模型输入边界。
- 决定：将官方 SP 作为独立观察保存；V1 模型仍只使用欧赔、亚洲让球、大小球及其多公司变化。
- 理由：保留未来研究价值，同时维持当前纯多公司市场基线的可解释边界。
- 影响：任何使用竞彩 SP 的模型实验都需要新增决策，不得静默加入特征。
- 替代方案：完全不保存官方 SP；会永久丢失前瞻时间线，未采用。

## D-015：提前进入阶段 3 标准化数据库

- 日期：2026-07-16
- 状态：已接受
- 背景：采集器单次验证和连续运行中期证据已经证明数据契约可落盘，项目负责人决定不再等待完整 7 天和 30 天周期后才建设分析库。
- 决定：允许立即实施阶段 3 PostgreSQL schema、迁移、可重放导入和 as-of 查询；阶段 1、2 验证继续并行运行。
- 理由：数据库层可以从不可变文件事实重建，提前建设不会改变原始数据，也能尽早暴露字段和查询问题。
- 影响：该决定覆盖 D-011、D-013 和执行计划中“验收完成后才进入数据库”的时间顺序，但不把任何未完成验收视为通过；采集器仍不得标记为生产可用，模型和 Web 门禁不解除。
- 替代方案：等待 30 天验收后再建库；项目负责人明确不采用。

## D-016：准备迁移到阿里云杭州单机 ECS

- 日期：2026-07-16
- 状态：已接受
- 背景：本地采集和 PostgreSQL 数据库已经形成可运行闭环，项目负责人希望评估迁移到阿里云服务器以减少本地机器长期运行约束。
- 决定：准备迁移到阿里云杭州 `cn-hangzhou` 单台 ECS，首版使用 Ubuntu 24.04 LTS x86_64、2 vCPU / 4 GiB、40 GB 系统盘、100 GB ESSD PL0 数据盘，同机运行采集器和 PostgreSQL 17；备份采用私有 OSS 与本地 G 盘迁移副本。当前不部署 RDS、Docker、Web、域名或负载均衡。
- 理由：单机 ECS 足以支撑当前单进程采集和可重建分析库，部署复杂度低；杭州地域便于使用 OSS 内网 Endpoint 和国内网络环境验证。
- 影响：需要实现精确窗口报告、Linux/systemd 运行、健康检查、OSS 风格内容寻址备份、云端 smoke test 和回滚规则。正式切换后 7 天和 30 天验收重新计时，本地与云端运行时间不得拼接。该决定部分替代 D-007 的地区假设和 D-013 的“首月仅本地运行”假设，但不解除采集器生产、模型或 Web 门禁。
- 替代方案：继续本地运行到 30 天验收后再迁移；直接使用 RDS/Docker/负载均衡；暂不采用。
