## D-001：V1 只使用盘口市场数据

- 日期：2026-07-15
- 状态：已接受
- 背景：需要验证仅依靠市场信息能否形成可校准、可复现的概率预测。
- 决定：V1 输入限定为欧赔、亚洲让球、大小球、多公司盘口变化和相关时间信息。
- 理由：控制变量和数据范围，优先解决数据质量、时间语义及防泄漏问题。
- 影响：球队新闻、伤停、阵容、天气、舆情和人工观点不进入 V1 模型。
- 替代方案：多源球队基本面模型；留待 V1 基线稳定后另行决策。

## D-002：预测口径为常规时间 90 分钟及补时

- 日期：2026-07-15
- 状态：已接受
- 背景：杯赛可能包含加时和点球，必须统一标签口径。
- 决定：所有预测和结算默认只计算常规时间 90 分钟及补时。
- 理由：与主流 1X2 常规时间市场保持一致，避免混合不同结算规则。
- 影响：加时和点球不得用于胜平负标签；赛果源必须能区分常规时间结果。
- 替代方案：预测晋级结果；不属于当前产品范围。

## D-003：使用四个固定预测切点

- 日期：2026-07-15
- 状态：已接受
- 背景：需要比较不同临场阶段并冻结可追溯版本。
- 决定：使用 `T-24h`、`T-6h`、`T-60m`、`T-10m` 四个切点。
- 理由：兼顾早期判断、赛前变化和临场状态。
- 影响：采集频率、新鲜度、快照和回测均围绕四个切点设计。
- 替代方案：连续实时预测；首版复杂度过高，暂不采用。

## D-004：原始数据 append-only 并保留四类时间

- 日期：2026-07-15
- 状态：已接受
- 背景：供应商可能更新或事后修正盘口记录，需要保留完整历史。
- 决定：原始响应只追加，并保存 `source_event_time`、`observed_at`、`ingested_at`、`corrected_at`。
- 理由：支持审计、重放和识别事后修正。
- 影响：供应商修正不能覆盖原始记录或已经发布的预测。
- 替代方案：只保存最新状态；无法可靠复现，拒绝采用。

## D-005：采用严格的 observed-at 防泄漏规则

- 日期：2026-07-15
- 状态：已接受
- 背景：数据源声明时间可能早于系统实际获得数据的时间。
- 决定：预测和回测只能使用 `observed_at <= prediction_cutoff` 的数据。
- 理由：确保离线回测不会使用当时系统尚未看到的信息。
- 影响：所有快照、特征、公司权重和训练切分都必须接受时间审计。
- 替代方案：只按供应商事件时间过滤；存在未来数据泄漏风险，拒绝采用。

## D-006：MVP 基础技术栈

- 日期：2026-07-15
- 状态：已接受
- 背景：项目需要可维护的采集、建模、API 和可视化能力。
- 决定：MVP 采用 Python、FastAPI、PostgreSQL、独立采集 Worker、React、ECharts 和 Docker Compose。
- 理由：技术成熟，适合单用户私有系统，并能覆盖数据和 Web 工作流。
- 影响：暂不默认引入 TimescaleDB、Redis 或 Celery，达到明确需求后再决策。
- 替代方案：单体脚本或更复杂的分布式架构；均不作为 MVP 默认方案。

## D-007：预算、部署地区和告警方式

- 日期：2026-07-15
- 状态：暂定
- 背景：执行计划需要一个可估算的运行环境，但尚未获得最终人工确认和供应商报价。
- 决定：暂按每月不超过 CNY 300、香港或新加坡 Ubuntu VPS、email 告警规划。
- 理由：为早期方案提供保守默认值。
- 影响：这些值不能作为采购授权，也不能用于降低数据验收标准。
- 替代方案：本地常驻、其他云地区或其他告警渠道；获得实际需求后再比较。

## D-008：当前采用单总控 Agent

- 日期：2026-07-15
- 状态：已接受
- 背景：项目仍在数据源调研期，立即维护多个角色 Agent 会增加交接和同步成本。
- 决定：使用根目录 `AGENTS.md` 作为单总控 Agent 契约，按项目阶段切换数据审计、数据工程、模型验证和产品运维职责。
- 理由：保持唯一状态入口和阶段门禁，同时为后续拆分保留清晰职责边界。
- 影响：当前不创建角色专属 Agent 文件或自定义 Skill。
- 替代方案：立即建立多角色 Agent；待并行开发需求实际出现时重新评估。
