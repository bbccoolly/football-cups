# 足球盘口预测平台长期执行计划

> 版本：V1.2
> 状态：执行计划
> 更新日期：2026-07-15
> 对应产品文档：`docs/product-plan.md`

## 1. 计划目标

本计划用于指导一个长期、可暂停、可恢复的足球盘口预测项目。项目目标是建立一个只使用盘口数据的赛前预测、临场监控和赛后复盘系统，覆盖五大联赛、欧战、世界杯及预选赛、韩国 K1 联赛。

本计划强调三件事：

- 每一步都要留下可追溯记录，方便间隔很久后继续。
- 数据源和时间语义先验收，再进入模型和产品开发。
- 所有预测、回测和评估都必须避免未来数据泄漏。

## 2. 固定项目规则

### 2.1 预测边界

所有预测默认针对常规时间 90 分钟及补时，不包含加时赛和点球大战。

系统不输出投注金额、投注组合、成本、收益或回报承诺。

### 2.2 数据边界

V1 阶段只使用盘口市场数据：

- 欧赔胜平负
- 亚洲让球盘
- 大小球盘
- 多公司盘口变化
- 数据源提供的时间戳和系统观察时间

不使用球队新闻、伤停、阵容、天气、舆情或人工主观判断作为模型输入。

### 2.3 时间边界

每条原始数据至少记录四个时间字段：

| 字段 | 含义 |
| --- | --- |
| `source_event_time` | 数据源声明的变盘或事件时间 |
| `observed_at` | 本系统首次看到该记录的时间 |
| `ingested_at` | 本系统写入数据库的时间 |
| `corrected_at` | 数据源事后修正时间，没有则为空 |

任何预测和回测只能使用 `observed_at <= prediction_cutoff` 的数据。

### 2.4 固定预测切点

每场比赛固定生成四个冻结预测版本：

| 切点 | 含义 |
| --- | --- |
| `T-24h` | 赛前一天的市场判断 |
| `T-6h` | 临近比赛日的稳定判断 |
| `T-60m` | 赛前一小时判断 |
| `T-10m` | 临场最终判断 |

已发布的预测版本不可覆盖。若发现数据或模型问题，只能创建新版本并记录原因。

## 3. 长期恢复协议

由于本项目可能每一步间隔很久，任何阶段开始前都必须先做恢复检查。

### 3.1 每次恢复时先检查

1. 查看 `docs/product-plan.md` 和本文件。
2. 查看 `docs/project-status.md`，确认当前阶段、上次决策和下一个动作。
3. 查看 `docs/decision-log.md`，确认历史选择，尤其是数据源、数据库、模型门槛和部署方式。
4. 查看 Git 状态，确认是否有未提交或未推送内容。
5. 检查 `.env` 或云端密钥是否仍有效，不要把密钥写入 Git 或聊天记录。

### 3.2 每完成一个小阶段必须更新

每个阶段完成后至少更新：

- 当前阶段状态
- 已完成事项
- 已发现风险
- 下一步动作
- 需要人工确认的问题
- 相关文件或报告路径

建议维护以下文档：

| 文档 | 作用 |
| --- | --- |
| `docs/project-status.md` | 当前进度和下一步 |
| `docs/decision-log.md` | 重要决策记录 |

## 5. 阶段 1：数据源候选与评分

目标：先确认可合法、稳定、持续获取盘口数据，再写采集代码。

### 5.1 人工查找数据源

需要找 2 到 4 个候选授权盘口 API。每个候选至少记录：

- 官网和文档地址
- 价格和套餐限制
- 请求频率限制
- 是否覆盖五大联赛、欧战、世界杯、世界杯预选赛、K1
- 是否支持欧赔、亚洲让球、大小球
- 是否支持历史盘口变化，不只是当前赔率
- 是否提供稳定的比赛 ID、赛事 ID、球队 ID、公司 ID
- 时间字段含义和时区
- 是否允许本地保存、分析、建模和长期使用
- 服务条款链接
- 退款、试用、技术支持方式

同时找两个独立校验源：

- 一个公开赔率展示页面，用于人工对比盘口。
- 一个独立比赛结果源，用于校验赛果。

公开网页只作为人工校验，除非其条款明确允许自动化采集。

### 5.2 数据源评分表

每个候选数据源按 100 分评分：

| 项目 | 分值 | 说明 |
| --- | ---: | --- |
| 授权和保存权利 | 25 | 是否明确允许保存、分析和长期使用 |
| 覆盖范围 | 20 | 是否覆盖目标赛事和目标盘口 |
| 历史变盘完整度 | 20 | 是否提供完整时间线，而不是少量快照 |
| 稳定 ID 和时间语义 | 10 | ID 是否稳定，时间字段是否清晰 |
| 可用性 | 10 | SLA、频率限制、响应速度、稳定性 |
| 价格 | 10 | 是否适合长期个人项目 |
| 文档和支持 | 5 | 文档清晰度、示例、技术支持 |

评分结论：

- `>= 80`：可以进入试用验证。
- `65-79`：条件通过，只能作为备选或降级源。
- `< 65`：淘汰。

### 5.3 人工浅层验证

至少人工检查 40 场比赛：

| 类型 | 数量 |
| --- | ---: |
| 五大联赛 | 10 |
| 欧战 | 10 |
| 世界杯或预选赛 | 10 |
| 韩国 K1 | 10 |

每场记录：

- 数据源比赛名称、开球时间、主客队
- 独立结果源的比赛名称、开球时间、主客队
- 至少一家公司的欧赔、让球、大小球是否存在
- 是否存在明显球队映射错误
- 是否存在时区偏差
- 是否存在盘口类型缺失

### 5.4 人工深层验证

至少深查 12 场比赛，每类 3 场：

- 五大联赛 3 场
- 欧战 3 场
- 世界杯或预选赛 3 场
- K1 3 场

每场检查：

- 2 到 3 家公司的完整变盘时间线
- 初盘、即时盘、封盘是否能识别
- 欧赔、让球、大小球的时间是否能对齐
- 公开展示源上的关键盘口是否大体一致
- 数据源是否会事后修正历史记录

### 5.5 自动化验证

试用 API 后，自动抽取至少 500 场目标比赛：

| 类型 | 数量 |
| --- | ---: |
| 五大联赛 | 200 |
| 欧战 | 100 |
| K1 | 100 |
| 世界杯或预选赛 | 100 |

自动检查：

- 核心字段完整率不低于 `99%`
- 三类盘口覆盖率不低于 `95%`
- 重复记录低于 `0.1%`
- 时间倒序记录低于 `0.1%`
- 稳定 ID 可跨接口关联
- 同一比赛的欧赔、让球、大小球可以按时间生成 as-of 快照

### 5.6 完成标准

- 产出 `docs/data-source-evaluation.md`
- 明确首选数据源、备选数据源和淘汰数据源
- 明确是否需要云端常驻采集器补齐历史变盘
- 未通过数据源验收前，不进入正式模型开发

## 6. 阶段 2：数据契约与数据库

目标：把盘口数据保存成可追溯、可回放、可建模的结构。

### 6.1 人工要确认

1. 数据源字段含义。
2. 数据源盘口类型枚举。
3. 数据源时间字段和时区。
4. 比赛状态枚举：未开、进行中、完场、延期、取消、中断。
5. 杯赛是否中立场、是否两回合、是否存在加时点球。

### 6.2 自动化要实现

1. PostgreSQL 表结构。
2. 原始响应 append-only 保存。
3. 赛事、赛季、球队、比赛、公司映射。
4. 欧赔、让球、大小球统一标准化。
5. 数据质量事件表。
6. 固定预测切点的 as-of 快照生成逻辑。

### 6.3 建议核心表

- `competitions`
- `seasons`
- `teams`
- `fixtures`
- `bookmakers`
- `raw_market_events`
- `market_events`
- `market_snapshots`
- `predictions`
- `prediction_scores`
- `results`
- `evaluations`
- `model_versions`
- `data_quality_events`

### 6.4 完成标准

- 能导入至少 500 场验证比赛。
- 能从原始记录重新生成标准化记录。
- 能按 `T-24h`、`T-6h`、`T-60m`、`T-10m` 生成快照。
- 能证明回测不会读取预测切点之后的数据。

## 7. 阶段 3：采集器与运行策略

目标：如果数据源没有完整历史变盘，就用常驻采集器从现在开始积累数据。

### 7.1 采集频率

| 距离开球 | 建议频率 |
| --- | --- |
| 72h 到 24h | 每 60 分钟 |
| 24h 到 6h | 每 30 分钟 |
| 6h 到 1h | 每 10 分钟 |
| 最后 1h | 每 2 分钟 |

### 7.2 快照新鲜度要求

| 预测切点 | 最大允许滞后 |
| --- | --- |
| `T-24h` | 2 小时 |
| `T-6h` | 30 分钟 |
| `T-60m` | 10 分钟 |
| `T-10m` | 3 分钟 |

超过新鲜度限制时：

- 降低置信度；
- 标记数据过期；
- 严重时停止发布该切点预测。

### 7.3 自动化要实现

1. 定时获取未来 72 小时比赛。
2. 按比赛开球时间动态调整采集频率。
3. 写入原始响应和标准化事件。
4. 采集失败自动重试。
5. 失败次数超阈值发送 email。
6. 每日备份数据库。
7. 每月生成数据质量摘要。

### 7.4 人工要做

1. 每周查看一次采集器是否正常。
2. 每月抽查至少 20 场比赛，对比公开展示源。
3. 每季度检查数据源条款是否变化。
4. 每季度恢复测试一次数据库备份。

### 7.5 完成标准

- 采集器可以无人值守运行 7 天。
- 失败、限流、空数据、字段变化都有日志。
- 数据库备份可恢复。
- 至少一封测试告警邮件可成功收到。

## 8. 阶段 4：离线基线模型

目标：先建立纯市场基线，确保系统不是靠复杂模型掩盖数据问题。

### 8.1 自动化要实现

1. 欧赔去水概率。
2. 多公司稳健中位数市场共识。
3. 亚洲让球和大小球标准化。
4. 由欧赔、让球、大小球联合反演主客进球参数。
5. 使用 Dixon-Coles 或相近方法生成比分矩阵。
6. 输出胜平负概率、市场隐含进球、三个比分点、总进球区间、双方进球概率。

### 8.2 回测要求

必须使用时间滚动验证：

```text
历史训练区间 -> 后续验证区间 -> 再向前滚动
```

禁止随机拆分。

每个赛事和每个预测切点分别报告：

- Log Loss
- Brier Score
- RPS
- ECE
- 可靠性曲线
- 比分 Top-1 和 Top-3 命中率

### 8.3 上线门槛

- 任意预测可由原始数据、截止时间和模型版本复现。
- 与同时间点市场基线相比，Log Loss 非劣化幅度不超过 `0.5%`。
- 整体 ECE 不高于 `0.03`。
- 使用 bootstrap 给出 `95%` 置信区间。
- 分赛事结果不能被高样本联赛掩盖。

### 8.4 完成标准

- 至少完成英超和 K1 的离线回测。
- 产出第一版模型报告。
- 明确基线模型的优势、失败样本和不适用场景。

## 9. 阶段 5：增强模型

目标：在有足够样本后，尝试学习盘口变化信息。

### 9.1 启动条件

满足以下条件后才启动 LightGBM 或其他增强模型：

- 至少 3,000 场符合条件的历史比赛。
- 每场至少包含一个有效预测切点。
- 三类市场覆盖率达标。
- 时间字段通过防泄漏检查。
- 基线模型已经稳定可复现。

### 9.2 可用特征

仅使用预测切点之前的数据：

- 初盘到当前盘口的概率变化
- 让球升降
- 大小球升降
- 水位变化幅度
- 变化速度
- 变化次数
- 回撤幅度
- 公司间分歧
- 三类市场一致性
- 距离开球时间
- 赛事、赛季阶段、是否中立场

### 9.3 禁止特征

- 赛后结果
- 开球后的盘口
- 全量数据计算出的公司权重
- 未来才知道的阵容、伤停或赛程事件
- 任何人工事后标注的结论

### 9.4 完成标准

- 增强模型必须和同时间点市场基线比较。
- 如果样本不足或收益不稳定，继续使用基线模型。
- 模型版本必须有独立 model card。

## 10. 阶段 6：私有 Web MVP

目标：做一个单用户、私有、可用于日常查看和复盘的系统。

### 10.1 技术选型

MVP 默认技术栈：

- Python
- FastAPI
- PostgreSQL
- 独立采集 Worker
- React
- ECharts
- Docker Compose

暂不默认引入 TimescaleDB、Redis、Celery，等数据量或并发真的需要再扩展。

### 10.2 页面

| 页面 | 功能 |
| --- | --- |
| 比赛中心 | 按日期、赛事、方向、置信度、风险筛选 |
| 单场分析 | 盘口时间线、公司分歧、比分矩阵、四个预测版本 |
| 临场监控 | 异动、冲突、方向变化和告警 |
| 预测档案 | 查看冻结预测版本 |
| 检查复盘 | 自动结算、偏差分析、异常标记 |
| 模型仪表盘 | 回测、校准、样本量、模型版本 |
| 数据质量中心 | 缺失、延迟、重复、映射冲突、采集失败 |

### 10.3 API 验收

至少提供：

- `GET /fixtures`
- `GET /fixtures/{id}`
- `GET /fixtures/{id}/markets`
- `GET /fixtures/{id}/predictions`
- `GET /fixtures/{id}/evaluation`
- `GET /model-versions`
- `GET /data-quality/events`

### 10.4 完成标准

- 能查看未来比赛。
- 能查看单场盘口历史。
- 能查看四个切点预测。
- 能查看赛后复盘。
- 能区分正常预测、降级预测和停止发布。

## 11. 阶段 7：扩展赛事

目标：在基础能力稳定后扩大覆盖范围。

### 11.1 扩展顺序

建议顺序：

1. 英超和 K1
2. 五大联赛全部覆盖
3. 欧冠、欧联、欧协联
4. 世界杯预选赛
5. 世界杯正赛

### 11.2 额外处理

欧战需要处理：

- 两回合
- 中立场
- 客场进球规则历史变化
- 淘汰赛常规时间和晋级目标不同

世界杯和预选赛需要处理：

- 国家队样本少
- 中立场或半中立场
- 洲际预选赛差异
- 小组赛末轮动机变化无法由纯盘口直接解释，只能由市场吸收

K1 需要处理：

- 公司覆盖可能少于五大联赛
- 盘口流动性较低
- 临场盘口可能更重要

### 11.3 完成标准

- 每个新增赛事先做数据覆盖报告。
- 每个新增赛事先通过基线回测。
- 样本不足时自动扩大不确定区间，不强行高置信输出。

- 检查是否有连续失败的数据源接口。

### 12.2 每月

### 12.3 每季度

- 人工复查数据源服务条款。
- 恢复测试一次数据库备份。
- 重新评分数据源。
- 复盘模型在各赛事的稳定性。

## 13. 阶段 1 启动入口

实时进度、阻塞项和唯一下一步统一记录在 `docs/project-status.md`，本节只保留阶段 1 的稳定启动规则，避免长期中断后按过期任务继续。

阶段 1 启动时需要人工完成：

1. 找 2 到 4 个授权盘口 API 候选。
2. 找 1 个独立公开赔率展示源。
3. 找 1 个独立赛果校验源。
4. 为每个 API 记录文档、价格、频率限制、覆盖范围、历史变盘、稳定 ID、时间语义、保存权利和条款链接。
5. 不要购买长期套餐，不要分享 API key。

获得候选信息后，Agent 继续完成：

1. 将证据录入 `docs/data-source-evaluation.md`。
2. 标记信息缺口和硬性淘汰项。
3. 根据候选信息评分，输出试用顺序和待人工确认问题。
4. 只有候选达到试用门槛且获得试用权限后，才编写范围受限的 API 验证脚本。
5. 数据源完整验收通过后，才进入阶段 2 设计数据库 schema。
