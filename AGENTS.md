# 项目总控 Agent 工作契约

本文件适用于整个仓库。项目默认使用中文沟通和编写文档，代码、字段、配置项及提交信息使用英文。

## 1. 启动与恢复顺序

每次开始工作时，依次执行：

1. 阅读本文件。
2. 阅读 `docs/product-plan.md`，确认产品边界。
3. 阅读 `docs/execution-plan.md`，确认阶段顺序和验收门槛。
4. 阅读 `docs/project-status.md`，确认当前阶段、阻塞项和下一步。
5. 阅读 `docs/decision-log.md`，确认已接受和暂定决策。
6. 检查 Git 状态以及本地与远程分支差异，保护已有未提交改动。

不得依赖聊天历史恢复项目状态。若聊天内容与仓库文档冲突，以仓库中已确认且日期较新的记录为核对起点，并先消除冲突。

## 2. 当前阶段门禁

当前阶段为“阶段 1：数据源候选与验收”。在 `docs/data-source-evaluation.md` 明确记录首选数据源通过验收前：

- 不开发正式数据库 schema、采集器、模型或 Web 产品。
- 可以维护治理文档、整理候选源证据、准备验证模板，以及在取得试用权限后编写范围受限的验证工具。
- 不购买长期 API 套餐，不假定任何候选源已经获得保存、分析或建模授权。

进入下一阶段必须同时满足执行计划的完成标准，并更新项目状态和决策日志。

## 3. 不可变业务与数据规则

- V1 只使用欧赔、亚洲让球、大小球、多公司盘口变化及其时间信息。
- 预测目标仅为常规时间 90 分钟及补时，不含加时赛和点球大战。
- 不输出投注金额、投注组合、成本、收益、ROI 或回报承诺。
- 原始数据 append-only，不覆盖供应商原始记录。
- 每条原始市场记录保留 `source_event_time`、`observed_at`、`ingested_at`、`corrected_at`。
- 预测和回测只能使用 `observed_at <= prediction_cutoff` 的记录。
- 已发布预测不可覆盖；修正必须形成带原因的新版本。

修改以上规则属于产品级决策，必须先更新产品方案和决策日志。

## 4. 人工与 Agent 分工

人工负责：

- 注册账号、申请试用、付费和接受服务条款。
- 阅读并确认授权、保存、分析、建模和长期使用权利。
- 在公开页面进行赔率、比赛信息和赛果抽查。
- 提供无法通过仓库或公开文档获得的账号内信息，但不得发送真实密钥。

Agent 负责：

- 整理证据、维护模板、执行评分并明确证据不足项。
- 在获得授权和测试凭据后编写采集验证、质量统计及报告工具。
- 检查时间语义、字段完整率、覆盖率、重复、倒序和映射问题。
- 维护当前状态、决策、风险、验收结果和下一项人工行动。

Agent 不得代替人工接受条款、购买服务，也不得根据营销页面推断数据使用权。

## 5. 密钥与本地数据

- 真实密钥只存入未跟踪的 `.env` 或云端密钥服务。
- 不得在代码、文档、Git、日志、测试快照或聊天中写入真实密钥、密码和连接串。
- 原始数据、数据库、备份和模型产物不得提交到 Git。
- 可提交的测试 fixture 必须体积小、已脱敏，并确认数据许可允许再分发。
- `.env.example` 只保留变量名和非敏感占位值。

## 6. 文档权威范围与冲突处理

- `docs/product-plan.md`：产品范围、输出和验收原则。
- `docs/execution-plan.md`：阶段顺序、任务和阶段完成标准。
- `docs/project-status.md`：当前进度的唯一入口，不是设计规范。
- `docs/decision-log.md`：决策历史；显式标记为“替代”的新决定取代旧决定。
- `docs/data-source-evaluation.md`：数据源证据、评分和验收结论。

发现文档、代码或外部证据冲突时，停止受影响的实现，记录冲突，核对原始证据并修正文档。不得静默选择一个版本继续。

## 7. 每次任务的完成条件

结束工作前必须：

1. 运行与改动风险相匹配的校验或测试。
2. 更新 `docs/project-status.md` 中的完成项、风险、阻塞项和唯一下一步。
3. 新增或改变重要选择时更新 `docs/decision-log.md`。
4. 确认没有密钥、本地数据或无关文件进入改动。
5. 检查 Git 状态，明确哪些内容已提交、未提交或未推送。
6. 向用户说明自动完成内容和仍需人工完成内容。
