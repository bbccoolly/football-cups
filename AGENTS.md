# 项目总控 Agent 工作契约

本文件适用于整个仓库。项目默认使用中文沟通和编写文档，代码、字段、配置项及提交信息使用英文。

## 1. 启动与恢复顺序

每次开始工作时依次阅读：

1. `AGENTS.md`
2. `docs/product-plan.md`
3. `docs/execution-plan.md`
4. `docs/project-status.md`
5. `docs/decision-log.md`
6. Git 状态和本地与远程分支差异

不得依赖聊天历史恢复项目。发现文档、代码或外部证据冲突时，先记录和消除冲突，不得静默选择一个版本继续。

## 2. 当前阶段门禁

当前开发阶段为“阶段 3：标准化数据库”。根据 D-015，项目负责人已授权在 500 连续验证尚未完成时提前建设分析数据库；这属于门禁覆盖，不代表 24 小时、7 天或 30 天验收已经通过。

当前允许：

- 继续运行并维护比赛发现、原始响应、标准化 JSONL、质量检查和验证采集器。
- 使用 SQLite 保存可重建的采集调度状态。
- 建立 PostgreSQL schema、迁移、JSONL/manifest 可重放导入、质量审计和 as-of 查询。
- 维护盘口标准化 V2、离线重放、采集资格与模型字段完整性分层，以及 V2 子窗口验收。
- 在独立 `data/research/` 和 PostgreSQL `research` schema 中导入公开静态历史数据，运行 D-022 允许的离线研究基线。
- 并行完成 24 小时、7 天和 30 天采集验证。
- 准备阿里云迁移文档、Linux/systemd 适配、健康检查、精确窗口报告和可恢复备份接口。
- 在隔离目录执行云端 smoke；正式 Linux timer 必须要求 `/srv/football-cups` 是真实数据盘挂载点。

当前仍禁止：

- 将验证采集器描述为长期生产可用。
- 在阶段 4 所需的至少 500 场严格快照及已验证赛果门禁完成前开发模型。
- 将 D-022 的历史研究基线用于当前比赛预测、正式模型、Web/API 或阶段 4 门禁计数。
- 在阶段 4 市场基线和阶段 5 增强模型门禁完成前开发 Web 产品。

PostgreSQL 是可重建分析层，不得成为原始事实来源。当前数据源用于个人学习研究；付费或授权 API 调研是未来可选项。

## 3. 不可变业务与数据规则

- 采集范围是 500 足球竞彩各玩法页面出现的全部比赛，不使用联赛白名单。
- V1 模型输入仍限定为欧赔、亚洲让球、大小球、多公司盘口变化和相关时间信息；竞彩官方 SP 先保存但不进入 V1 模型。
- 预测和标签只计算常规时间 90 分钟及补时，不包含加时赛和点球大战。
- 不输出投注金额、投注组合、成本、收益、ROI 或回报承诺。
- 原始响应 append-only，不覆盖历史记录；同内容可按 SHA-256 去重，但每次观察必须保留独立 manifest。
- 每条市场记录保留 `source_event_time`、`observed_at`、`ingested_at`、`corrected_at`。
- `observed_at` 是 HTTP 响应完整接收时间；预测和回测只能使用 `observed_at <= prediction_cutoff` 的数据。
- `SnapshotBatch.strict_eligible` 只表示采集时间、身份和时钟资格；模型资格还要求欧赔、亚盘、大小球均有已接受的 V2 标准化，且每个市场至少 3 家不同的字段完整 `bookmaker`。
- 盘口标准化修复只能追加 `normalization_version=2` 记录并保留来源哈希；历史离线重放不得访问网络、覆盖 V1、伪造切点或进入实时 7 天/30 天指标。
- 历史回抓统一标记 `backfill=true`、`strict_backtest_eligible=false`。
- 历史研究记录还必须标记 `research_only=true`、`cutoff_eligible=false`，不得映射为正式预测切点。
- 已验证赛果和已发布预测不可覆盖；冲突或修正必须形成带原因的新版本。
- 赛果闭环默认采用全自动确认与保守隔离；根据 D-028，项目负责人也可直接声明现有唯一候选属于 90 分钟结果。人工声明只能补充口径，不能输入或修改比分，必须追加为 `project-owner-manual-declaration`、保留候选引用并与自动指标分离。
- 中国体彩官方竞彩足球赛果页可作为明确 90 分钟口径证据源；必须先保存官方 scope、清单、fixture 映射和详情一致性证据。未完成精确映射、分页完整性、scope 证明或详情一致性时只能候选或隔离，不能自动验证。官方补偿默认每日最多一次；567、CORS、验证码或不完整清单必须记为来源失败，不得伪装成 fixture 缺失。
- 人工可确认场次无效、取消或未结算，但只能追加 `fixture_invalidated` 排除证据，不得人工补写比分；原始响应、manifest、JSONL 和既有盘口记录仍不得删除。
- Windows 正式运行采用 D 盘事实层、G 盘双层异盘备份；每日镜像和每周内容寻址备份必须通过一致性快照，不能与活动 JSONL 写入并发。
- 历史研究只下载显式注册的公开静态文件；禁止批量访问 500 历史页面，禁止代理/IP/Cookie 轮换、指纹伪装、验证码绕过或其他身份规避。

修改以上规则属于产品级决策，必须同步更新产品方案和决策日志。

## 4. 人工与 Agent 分工

人工负责：

- 保证验证机器联网、不休眠并校准系统时间。
- 7 天验收期间每日核对页面比赛数量和至少 3 场解析结果。
- 长期运行后每周查看日报、每月抽查至少 20 场盘口和自动赛果分类。
- 保证 G 盘在线，并在注销和重启验收时确认盘符及物理磁盘映射未变化。
- 对无法自动证明口径但已自行核验的候选，可明确声明其属于常规时间；声明真实性由项目负责人负责。

Agent 负责：

- 维护发现、采集、标准化、调度、质量统计、日报、备份和恢复工具。
- 将来源缺盘与程序失败分开统计。
- 对身份冲突、结构变化、时间漂移、切点迟到和赛果歧义生成质量事件。
- 维护乱码检测、盘口线转换、V2 标准化证据和模型资格原因；让球指数解析失败不得阻止三个核心市场的 V1 模型资格。
- 维护赛事格式登记；普通联赛可自动确认，可能加时或未知赛事自动隔离，不根据名称临时猜测。
- 维护中国体彩官方赛果证据链、低频自动补偿和引用完整性审计；500 未确认口径候选与官方 90 分钟比分不一致时保留两者并产生 `result_scope_difference`，只有明确 90 分钟来源之间冲突才阻止验证。
- 将项目负责人声明通过稳定命令追加为已验证结果；不得接收人工比分，批量预检失败时不得写入任何确认记录，人工覆盖率不得改善自动赛果验收指标。
- 维护备份新鲜度、任务日志、每月恢复演练和 Windows 无人值守任务状态。
- 每次结束前更新项目状态；产生新决策时更新决策日志。

## 5. 密钥、本地数据与外部项目

- 真实密钥只存入未跟踪的 `.env` 或系统密钥服务，不得进入代码、文档、Git、日志或聊天。
- 原始数据、SQLite、数据库、备份、日志和模型产物不得提交 Git。
- 可提交 fixture 必须体积小、已脱敏且仅用于离线测试。
- 主项目运行时不得依赖 `D:\2026-worldCup` 或其他仓库路径；可移植通用思路，但实现和测试必须完整存在于本仓库。

## 7. 每次任务的完成条件

1. 运行与改动风险匹配的测试或校验。
2. 更新 `docs/project-status.md` 的完成项、风险、阻塞和唯一下一步。
3. 新增或改变重要选择时更新 `docs/decision-log.md`。
4. 确认没有密钥、本地数据或无关文件进入改动。
5. 检查 Git 状态，说明提交、推送和本地运行状态。
