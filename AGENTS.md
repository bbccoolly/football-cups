# 项目总控 Agent 工作契约

本文件适用于整个仓库。项目默认使用中文沟通和编写文档，代码、字段、配置项及提交信息使用英文。

## 1. 启动与恢复顺序

每次开始工作时依次阅读：

1. `AGENTS.md`
2. `docs/product-plan.md`
3. `docs/execution-plan.md`
4. `docs/project-status.md`
5. `docs/decision-log.md`
6. Git 状态和本地与远程分支差异

不得依赖聊天历史恢复项目。发现文档、代码或外部证据冲突时，先记录和消除冲突，不得静默选择一个版本继续。

## 2. 当前阶段门禁

当前开发阶段为“阶段 3：标准化数据库”。根据 D-015，项目负责人已授权在 500 连续验证尚未完成时提前建设分析数据库；这属于门禁覆盖，不代表 24 小时、7 天或 30 天验收已经通过。

当前允许：

- 继续运行并维护比赛发现、原始响应、标准化 JSONL、质量检查和验证采集器。
- 使用 SQLite 保存可重建的采集调度状态。
- 建立 PostgreSQL schema、迁移、JSONL/manifest 可重放导入、质量审计和 as-of 查询。
- 并行完成 24 小时、7 天和 30 天采集验证。

当前仍禁止：

- 将验证采集器描述为长期生产可用。
- 在阶段 4 所需的至少 500 场严格快照及已验证赛果门禁完成前开发模型。
- 在阶段 4 市场基线和阶段 5 增强模型门禁完成前开发 Web 产品。

PostgreSQL 是可重建分析层，不得成为原始事实来源。当前数据源用于个人学习研究；付费或授权 API 调研是未来可选项。

## 3. 不可变业务与数据规则

- 采集范围是 500 足球竞彩各玩法页面出现的全部比赛，不使用联赛白名单。
- V1 模型输入仍限定为欧赔、亚洲让球、大小球、多公司盘口变化和相关时间信息；竞彩官方 SP 先保存但不进入 V1 模型。
- 预测和标签只计算常规时间 90 分钟及补时，不包含加时赛和点球大战。
- 不输出投注金额、投注组合、成本、收益、ROI 或回报承诺。
- 原始响应 append-only，不覆盖历史记录；同内容可按 SHA-256 去重，但每次观察必须保留独立 manifest。
- 每条市场记录保留 `source_event_time`、`observed_at`、`ingested_at`、`corrected_at`。
- `observed_at` 是 HTTP 响应完整接收时间；预测和回测只能使用 `observed_at <= prediction_cutoff` 的数据。
- 历史回抓统一标记 `backfill=true`、`strict_backtest_eligible=false`。
- 已验证赛果和已发布预测不可覆盖；冲突或修正必须形成带原因的新版本。

修改以上规则属于产品级决策，必须同步更新产品方案和决策日志。

## 4. 人工与 Agent 分工

人工负责：

- 保证验证机器联网、不休眠并校准系统时间。
- 7 天验收期间每日核对页面比赛数量和至少 3 场解析结果。
- 长期运行后每周查看日报、每月抽查至少 20 场盘口和赛果。
- 为新赛事登记 `regular_time_only`、`may_have_extra_time` 或 `unknown`。
- 在 30 天验收前提供另一物理磁盘或网络备份目录。
- 对杯赛等无法自动确认的 90 分钟赛果进行人工核验。

Agent 负责：

- 维护发现、采集、标准化、调度、质量统计、日报、备份和恢复工具。
- 将来源缺盘与程序失败分开统计。
- 对身份冲突、结构变化、时间漂移、切点迟到和赛果歧义生成质量事件。
- 每次结束前更新项目状态；产生新决策时更新决策日志。

## 5. 密钥、本地数据与外部项目

- 真实密钥只存入未跟踪的 `.env` 或系统密钥服务，不得进入代码、文档、Git、日志或聊天。
- 原始数据、SQLite、数据库、备份、日志和模型产物不得提交 Git。
- 可提交 fixture 必须体积小、已脱敏且仅用于离线测试。
- 主项目运行时不得依赖 `D:\2026-worldCup` 或其他仓库路径；可移植通用思路，但实现和测试必须完整存在于本仓库。

## 6. 文档权威范围

- `docs/product-plan.md`：产品范围、输出和长期验收原则。
- `docs/execution-plan.md`：阶段顺序、任务和阶段完成门槛。
- `docs/project-status.md`：当前进度和唯一下一步。
- `docs/decision-log.md`：决策历史。
- `docs/data-source-evaluation.md`：500 技术验收记录。
- `docs/500-collector-spec.md`：采集器稳定数据与行为契约。
- `docs/collector-runbook.md`：本地运行、备份和恢复操作。
- `docs/database-design.md`：PostgreSQL schema、导入和 as-of 查询契约。
- `docs/database-runbook.md`：本地 PostgreSQL 安装、迁移、重放和恢复操作。

## 7. 每次任务的完成条件

1. 运行与改动风险匹配的测试或校验。
2. 更新 `docs/project-status.md` 的完成项、风险、阻塞和唯一下一步。
3. 新增或改变重要选择时更新 `docs/decision-log.md`。
4. 确认没有密钥、本地数据或无关文件进入改动。
5. 检查 Git 状态，说明提交、推送和本地运行状态。
